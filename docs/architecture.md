# Gamarr Architecture

## Overview

Gamarr is a server-first deployment platform for user-supplied PC game installation media. The server owns package metadata, package definitions, machine registration, jobs, state transitions, events, and logs. Windows agents execute work on target machines and report progress back to the server.

## Current Hardening and Package Definition Decisions

- PostgreSQL is the source of truth for packages, machines, jobs, events, and logs.
- RabbitMQ is a durable dispatch signal, not the authoritative work queue.
- Agents use HTTP for registration, heartbeat, polling, claim, and event/log reporting.
- Each package version now persists both:
  - normalized relational fields for querying and UI rendering
  - an immutable manifest artifact (`ManifestJson`) generated by the server
- Execution remains safe and constrained in the current slice:
  - `MockRecipe` execution is supported
  - arbitrary PowerShell execution is not enabled yet

## Components

### API server

- ASP.NET Core Web API
- persists authoritative state in PostgreSQL
- exposes package, machine, job, and agent endpoints
- applies startup migrations and optional demo seeding
- publishes RabbitMQ notifications when jobs are created or updated
- validates requests and returns problem-details style errors for common failures
- generates canonical package manifests from package create requests

### Web UI

- React + TypeScript SPA
- arr-style self-hosted control surface
- dashboard, package library, package detail, machines, jobs, and detail views
- package detail now renders:
  - version metadata
  - media definitions
  - script references
  - detection rules
  - prerequisites
  - raw manifest artifact

### Agent

- .NET worker / Windows service
- stable machine identity persisted locally
- machine registration and heartbeat
- polling for next available job
- explicit claim before execution
- safe mock recipe execution with ordered state and log reporting
- transient failure handling with retry/backoff
- detection rule evaluation during validation

### PostgreSQL

- authoritative store for:
  - packages
  - package versions
  - manifest artifacts
  - media references
  - detection rules
  - prerequisites
  - machines and capabilities
  - jobs
  - job events
  - action logs

### RabbitMQ

- durable dispatch signal only
- emits `job.created` and `job.updated`
- not used as the source of truth for work ownership

## Package Definition Model

### Package and version

- `Package`: stable catalog item
- `PackageVersion`: deployable version metadata plus execution definition

`PackageVersion` now includes:

- version label
- supported OS
- architecture
- install script kind
- install script path
- uninstall script path
- timeout
- notes
- manifest format version
- manifest JSON artifact

### Media and validation

- `PackageMedia`: local or UNC media reference
- `InstallDetectionRule`: validation rule for installed state
- `PackagePrerequisite`: explicit prerequisite metadata

## Manifest Format

Current manifest format:

- `gamarr.package/v1`

Manifest contents:

- package identity and metadata
- package version metadata
- script definitions
- media definitions
- detection rules
- prerequisites

The server generates and persists the canonical manifest artifact from package create input. This keeps the current API/UI/query model stable while creating a durable package-definition format for future import/export workflows.

## Execution Model

### Current supported execution kind

- `MockRecipe`

The agent supports a constrained mock recipe file format for safe local execution. It currently understands:

- `LOG <message>`
- `WAIT <seconds>`
- `WRITE_FILE <relativePath>|<content>`

Writes are constrained to an agent-controlled mock install root under ProgramData. Detection rules can reference `%MOCK_INSTALL_ROOT%` and are expanded by the agent during validation.

### Future execution kind

- `PowerShell`

The domain and API now carry the script kind explicitly, but the agent does not execute arbitrary PowerShell in this slice.

## Job Lifecycle

The persisted job state machine is:

- `Queued`
- `Assigned`
- `Preparing`
- `Mounting`
- `Installing`
- `Validating`
- `Completed`
- `Failed`
- `Cancelled`

Rules enforced in code:

- claim moves `Queued -> Assigned`
- the agent emits ordered transitions beginning at `Preparing`
- event sequence numbers must be contiguous
- terminal jobs reject further mutation
- machine status flips to `Busy` when work is claimed and returns to `Online` on terminal completion

## Current Limitations

- no real auth yet
- no file upload/import workflow yet
- no real ISO mounting yet
- no real PowerShell recipe execution yet
- no checksum verification or media existence validation yet
- no agent-side concurrent job execution
- no websocket/live push updates
- no verified runtime pass was possible in this shell environment

## Roadmap

### Near-term

- replace `MockRecipe`-only execution with real media validation and PowerShell execution behind the same executor seam
- add manifest import/export endpoints and version creation workflows
- support richer detection rule types and validation results
- improve install log capture and log streaming structure
- add server and agent test projects

### Mid-term

- add local-admin authentication and sessions
- add package import workflows for local folders and executables
- add richer machine diagnostics and compatibility reporting
- add retries, cancellation handling, and clearer failure reasons
- add CI validation for API, web, and agent contracts

### Later

- multi-version package management UX
- role-aware access control
- richer artifact validation and checksums
- websocket or signal-based live UI updates
- fleet-scale machine grouping and scheduling policies

## Best Next Implementation Milestone

Implement the real execution pipeline behind the now manifest-backed contracts:

- resolve and validate media paths
- mount ISOs when required
- invoke PowerShell package recipes for trusted definitions
- execute richer detection rules after install
- persist richer install logs and validation summaries
